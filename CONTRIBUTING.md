# Contributing

Thanks for considering a contribution to Memory Skill System.

This repository is not trying to be a generic note-taking framework.
It is trying to define a clearer operating model for memory work across ongoing projects and reusable methods.

---

## First principle

Keep the repository centered on these goals:
- structured recall
- continuation rather than recap
- safe routing across permission/domain/type
- durable write-back without context dumping
- clear boundary between memory and detailed project assets

If a change weakens those goals, it needs strong justification.

---

## Good contribution targets

Useful contributions usually improve one of these:
- routing clarity
- continuation quality
- public/shareable packaging quality
- documentation clarity
- boundary clarity with project assets and PM workflow

---

## Changes that need extra care

Please be more careful when changing:
- routing dimensions
- permission/domain/anchor/type meanings
- write-back rules
- shared-context safety rules
- relationship to PM workflow
- examples of what memory should and should not store

These are closer to the protocol layer of the skill.

---

## Contribution rules

### Rule 1: Keep memory distinct from project assets
Do not turn this repository into a substitute for detailed project baselines, demos, PRDs, or version packages.

### Rule 2: Avoid context mixing
Do not weaken the routing model in a way that casually mixes private, work, and reusable-method memory.

### Rule 3: Prefer usable clarity over abstraction for its own sake
Changes should make the system easier to apply, not just more conceptually elaborate.

### Rule 4: Keep public packaging public-safe
Do not introduce private personal assumptions, hidden local context, or unpublished sensitive examples into the public repository.

### Rule 5: Keep docs aligned with behavior
If routing logic, boundaries, or intended usage changes, update the related docs too.

---

## Pull request guidance

A good PR usually explains:
- what changed
- why it changed
- whether it affects routing, write-back, boundaries, or only wording
- whether related docs were updated
